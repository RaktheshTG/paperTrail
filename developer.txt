Basics :
Vite vs React - vite is the tool that runs react during development and execution stage.
                react is a js library that helps in building the ui.


index.html
touch index.html to change the page title, add a favicon, or add a Google Fonts link. 
That's it. it has div id = "root" where react mounts


TanStack Router
TanStack Router is a library that fakes page navigation inside React.
When you go to /about, it doesn't load a new file — it just swaps which
component is displayed. This is called "client-side routing".


router.tsx 
creates the router instance and tells it "here are all the routes in this app."


routes/ folder — each file in here IS a page. The file name = the URL path:

routes/index.tsx → shows at /
routes/about.tsx → would show at /about
routes/__root.tsx → wraps EVERY page (good for navbars, footers)


routeTree.gen.ts
TanStack auto-generates this by scanning your routes/ folder. Never touch it.


src/routes/index.tsx 
a React component that represents the homepage (/). This is where your entire
PaperTrail UI lives.


App.jsx?
There isn't one in this project. In classic React setups (Create React App),
App.jsx is the root component. Here, routes/__root.tsx plays that role — it wraps
every page. And routes/index.tsx is your main page component. TanStack Router
replaced the need for a separate App.jsx.


Shadcn and ui folder
shadcn gives you <Button>, <Tabs>, <Dialog>, <Card> etc. — pre-styled, accessible,
ready to use. The key difference from other UI libraries: the code lives in your
project, inside src/components/ui/. You can open src/components/ui/button.tsx and
change how every button looks across the whole app.

<button className="bg-purple-600 px-4 py-2 rounded-lg text-white hover:bg-purple-700">
  Click me
</button>

// you write:
<Button variant="default">Click me</Button> and go to ui folder and button to modify it


The lib/ folder
lib/ is for non-UI code — utilities, API connections, data helpers. 
Nothing in here renders anything on screen. Think of it as the "brain" folder
vs the "face" folder.

groq.ts — connects to Groq API. Pure logic, no UI.

demo-data.ts — hardcoded strings and objects used as placeholder content
 until real AI responses come in.

utils.ts — one tiny function called cn() that merges Tailwind class names 
cleanly. shadcn uses it everywhere.

ALLORIGINS CONCEPT :
The CORS problem is a browser security rule. When your React app (running at localhost:8080)
tries to fetch from ar5iv.org, the browser checks:

"Does ar5iv.org have a header saying it allows requests from localhost:8080?"

ar5iv says no → browser blocks it. This isn't ar5iv being mean — it's a default 
security policy. Servers have to explicitly opt in to allow cross-origin requests.
allorigins is a middleman:

Without proxy:
Browser → ar5iv.org ✗ BLOCKED

With allorigins:
Browser → allorigins.win → ar5iv.org ✓ (server to server, no CORS rules)
         ← allorigins wraps response in JSON ←
         ← browser receives it fine ←

Server-to-server requests don't have CORS restrictions — that's a browser-only rule.
allorigins is just a tiny server that fetches any URL you give it and returns the
content wrapped in JSON: ( thats whay res.json() not res.text()  and then data.contents to extract data )

RAG PIPELINE :
1. CHUNKING       — split the 80k character paper into small pieces (~500 tokens each)
2. EMBEDDING      — convert each chunk into a vector (list of numbers representing meaning)
3. STORAGE        — keep chunks + their vectors in memory (our "in-browser vector DB")
4. RETRIEVAL      — when user asks a question, embed the question too, 
                     find the chunks whose vectors are most "similar" to the question's vector
5. GENERATION     — send ONLY those relevant chunks (not the whole paper) to Groq for the answer

WHY RAG :
Right now we just slice the first 3000 characters and hope the answer is in there. 
If someone asks about something in section 5 of the paper, but we only sent section 1, 
Groq can't answer correctly. RAG fixes this by actually finding the relevant part of the 
paper for each specific question.



EMBEDDINGS :
An embedding turns text into a list of numbers (a vector) that captures meaning:
"The Transformer uses attention" → [0.23, -0.81, 0.44, ... ] (1536 numbers)
"Self-attention mechanism"        → [0.21, -0.79, 0.41, ... ] (very similar numbers!)
"What did you eat for breakfast"  → [0.91, 0.12, -0.55, ... ] (very different numbers)

To find "what's relevant to this question," we just measure the mathematical distance 
between vectors — this is called cosine similarity. ( NLP )

For generating embeddings (the vector conversion step), which approach?

Embeddings work perfectly. "Vector dimensions: 1024" confirms Cohere converted your text 
into a 1024-number list, and you can see the actual numbers.

Think about how you, a human, know that "happy" and "joyful" mean similar things, but "happy" and "table" don't. You understand meaning. Computers don't naturally understand meaning — they only understand numbers.
Embeddings solve this by converting text into a list of numbers (a vector) in such a way that:

Similar meanings → similar numbers
Different meanings → different numbers
After vector number calc -> use cosine similarity

Example 
"The cat sat on the mat"              → [0.024, 0.011, 0.025, -0.004, -0.032, ... 1024 numbers total]
"A feline rested on the rug"          → [0.021, 0.013, 0.024, -0.005, -0.030, ...] (VERY close numbers!)
"Attention mechanisms in transformers" → [0.91, -0.55, 0.12, 0.77, -0.23, ...] (totally different numbers)

Here's the real process (simplified):

Cohere took billions of sentences from the internet

They trained a neural network with a task: "predict which sentences are similar in 
meaning and which aren't" (using techniques like contrastive learning — showing the
model pairs of similar and dissimilar sentences millions of times)

During training, the network adjusts millions of internal weights so that when you feed 
text through it, the OUTPUT vector naturally places similar-meaning text close together

After training is done, that network becomes the "embedding model" — you feed it your 
sentence, it runs through all those learned weights, and outputs 1024 numbers
